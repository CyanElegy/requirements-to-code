# UI 参考输入与分析

需求文档通常只记录业务流程（"系统应该做什么"），不记录 UI 层信息（页面长什么样、按钮放哪里）。在功能树和架构确定后、原型生成前，需要主动收集 UI 参考来确定页面布局。

---

## 缺失的 UI 信息

| 缺失的 UI 信息 | 为什么重要 |
|---|---|
| 页面布局结构 | 筛选栏在顶部还是侧边？列表和详情左右分还是上下分？ |
| 交互模式 | 行点击是弹窗、抽屉还是新页面？操作按钮暴露在外还是收进"..."菜单？ |
| 信息层级 | 哪些字段在列表页就显示？哪些折叠到详情？ |
| 视觉风格 | 企业后台风格还是消费级风格？信息密度高还是低？ |
| 响应式策略 | 移动端是隐藏侧边栏还是折叠成汉堡菜单？ |

AI 凭空填补这些信息容易产生"AI 味"的设计——功能齐全但布局平庸、交互模式同质化。

---

## 五类参考输入（按优先级）

| 优先级 | 参考类型 | 分析方式 | 适用场景 |
|---|---|---|---|
| 1 | **参考截图**（PNG/JPG） | Read 工具读取图片，分析布局、组件、交互、层级 | 已有旧系统、竞品截图 |
| 2 | **参考 URL** | WebFetch 获取页面，分析 DOM 结构和视觉风格 | 有在线参考产品 |
| 3 | **Figma 设计稿** | Figma MCP 工具获取设计稿数据 | 有设计师交付的设计稿 |
| 4 | **口头描述** | 用户简单描述想要的大致布局 | 无截图但有经验直觉 |
| 5 | **无参考（默认）** | 根据功能树自动推断合理布局 | 完全从零开始 |

---

## 收集阶段的话术

在阶段 0 主动询问用户：

> 需求文档描述的是业务逻辑，但页面长什么样、按钮放哪里等信息通常不会写在里面。你可以提供以下任何一种来帮助我确定页面布局：
> 1. **参考截图** — 已有系统的页面截图（JPG/PNG），我会分析布局结构
> 2. **参考 URL** — 类似产品的线上页面
> 3. **Figma 链接** — 设计师交付的设计稿
> 4. **口头描述** — 比如"顶部搜索栏，左边分类树，右边卡片网格"
> 5. **暂无参考** — 我会根据功能树自动推导合理布局（默认）
>
> 不同页面可以有不同的参考来源，也可以只为一个页面提供参考。

## 截图分析法

1. **用 Read 工具读取图片**（Read 支持 PNG/JPG）
2. **分析布局骨架**：
   ```
   页面整体布局：[header + sidebar] | [header + content-only] | [header + sidebar + right-panel]
   导航位置：[top-nav] / [side-nav] / [top+side]
   内容区域划分：筛选区、主列表区、详情面板、操作栏
   ```
3. **识别 UI 组件**：表格（striped? bordered? density?）、表单（label 对齐? 单列/双列?）、筛选栏（水平/折叠/侧边?）、分页（位置? 功能?）
4. **提取交互模式**：行点击 → 弹窗 / 抽屉 / 新页面 / 展开行内详情；批量操作 → checkbox + 顶部操作栏；Tab 切换；面包屑导航
5. **分析信息层级**：页面标题位置和大小、主要操作按钮位置、次要操作按钮位置、数据密度判断

分析结果并非 1:1 复刻截图样式，而是提取其**布局骨架和交互模式**，填入新需求的**数据和功能**，生成新页面布局。

## URL/Figma 分析法

- **URL**：WebFetch 获取页面内容和截图，做同样的结构分析
- **Figma**：通过 Figma MCP 读取设计稿的组件树和布局约束
- **口头描述**：直接解析用户的自然语言描述，映射到布局模式

## 使用 frontend-design skill 推导设计（无参考时）

**当用户没有任何 UI 参考时，不要凭经验猜测布局。** 调用 `frontend-design` skill 来为每个页面推导有意的、独特的视觉设计。

### 为什么用 frontend-design skill

简单的"特征→布局"映射表（如"数据列表 = 表格+分页"）容易产生模板化的、缺乏辨识度的设计。frontend-design skill 提供：
- **有意的视觉方向** — 不只是选择布局模式，而是为整个系统建立一致的审美基调
- **排版与间距系统** — 超越默认的 16px/24px 思维
- **独特的配色方案** — 不是千篇一律的蓝色 sidebar + 白色内容区
- **避免 AI 味设计** — 功能齐全但布局平庸、交互模式同质化

### 调用方式

对每个页面（或一组风格统一的页面），使用 Skill 工具调用 `frontend-design` skill，传入：

1. **页面的功能描述**（来自 features.json 中该页面的功能节点）
2. **数据特征** — 有哪些字段、什么类型、数量级
3. **用户角色** — 谁在用这个页面、核心任务是什么
4. **整体产品定位** — 后台管理系统 / 消费级应用 / 数据仪表盘 / 工具型页面

让 `frontend-design` skill 输出：
- 该页面的布局模式选择 + 理由
- 配色与排版建议
- 组件交互模式建议
- 响应式策略

### 整合到 layout-reference.json

将 frontend-design skill 的输出映射为 layout-reference.json 的 zones、interactionPatterns、designTokens。如果涉及多个页面，确保设计一致性——颜色、字体、间距、组件风格在所有页面间统一。

### 参考推导表（作为 frontend-design skill 的输入提示）

以下映射表作为给 frontend-design skill 的功能上下文，不是最终布局决定：

| 页面特征 | 常见的布局起点（供 design skill 参考与突破） |
|---|---|
| 数据列表 + 筛选条件 > 3 个 | 顶部筛选栏 + 数据表格 + 分页（可突破：用侧边筛选面板） |
| 数据列表 + 左侧分类/树 | 左 sidebar 分类 + 右内容区 |
| 详情页（单实体） | 页头 + 分组信息卡片 + 操作栏 |
| 表单页（创建/编辑） | 居中窄表单或两栏布局 |
| Dashboard / 数据概览 | 统计卡片行 + 图表网格 |
| 设置页 | tab 导航 + 表单区 |

**每次推导时必须声明设计决策理由，让用户感知布局是有意为之而非随机选择。**

## 多页面参考映射

如果一个系统的截图覆盖多个页面类型，用户可以按页面映射：

```
/orders → 参考截图: legacy-orders-list.png（旧系统订单列表页）
/orders/:id → 参考截图: legacy-order-detail.png（旧系统订单详情页）
/users → 口头描述: "类似订单列表，但左侧有部门树"
/dashboard → 无参考，自动推导
```

## 产出：layout-reference.json

分析结果输出为 `specs/layout-reference.json`。完整 schema 见 `references/schemas.md`。

layout-reference.json 在下游的使用：
- **原型生成**：每个 zone 对应一个 DOM 区块；interactionPatterns 决定按钮行为和导航逻辑
- **代码生成**：zone 映射到组件树；responsiveStrategy 指导响应式实现
- **迭代模式**：layout 级别保持不变（除非用户说"改布局"），delta 只记录 layout 以上变更
