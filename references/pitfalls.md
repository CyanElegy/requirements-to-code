# 常见陷阱与恢复策略

## 1. 跳过状态处理

**症状**：原型或代码只实现了"正常"状态，loading/empty/error 被忽略。

**后果**：看似完成了功能，但实际上交付的是一个残缺的组件——用户在网络慢、数据为空、接口出错时看到的是空白页面或未处理的状态。

**恢复**：回顾 `specs/features.json` 中每个功能节点的 `states` 字段，逐项检查是否在原型和代码中有对应实现。每个数据展示组件必须有 loading（骨架屏）、empty（空状态引导）、error（错误+重试）、normal 四种状态的 UI。

---

## 2. 孤立交互

**症状**：按钮渲染了但 `onClick` 没有正确实现；链接的 `href` 指向不存在的路由；表单提交后没有反馈。

**后果**：原型看起来完整但点击无反应，用户/评审方对交互流程无法验证。

**恢复**：对照 `specs/routes.json` 和功能节点的 `interactions` 字段，逐一检查每个触发-效果对是否在原型/代码中实现。原型中的按钮至少要有关闭弹窗、切换 Tab、跳转锚点等可感知反馈。

---

## 3. 数据依赖模糊

**症状**：功能节点的 `dataDependencies` 中字段缺少类型、来源是猜测、没有标注 `inferred: true`。

**后果**：代码生成阶段基于不正确的数据假设生成类型定义和 API 调用；对接真实后端时大量返工。

**恢复**：重新审视需求文档中明确提到的数据结构，与模糊推断的字段分开。推断字段必须标注 `"inferred": true`，在 OpenAPI 中标注 `[推断字段，待确认]`，并列表给用户确认。

---

## 4. 忽略已有代码规范

**症状**：迭代模式下新代码使用了不同的状态管理方式、命名风格、文件组织方式，与现有代码风格不一致。

**后果**：代码库出现"风格断层"，维护成本增加。

**恢复**：在阶段 0 的项目扫描中，必须采样至少 3-5 个现有组件的代码风格（声明方式、类型位置、样式方案、状态管理、文件名规范、路径别名），产出 `specs/project-context.md`，后续所有阶段严格遵循。如果用户反馈"风格不对"，回到 project-context.md 补充采样再重新生成。

---

## 5. Mock 数据太简单

**症状**：Mock 数据只有 1-2 条、"test"、"foo bar" 等无意义内容。

**后果**：原型和测试无法展示真实场景；空状态、长文本溢出、边界情况等无法验证。

**恢复**：使用确定性工厂函数，生成至少 10-20 条数据。数据要真实：中文姓名、合理数值范围、真实日期分布、不同的状态组合。通过 URL 参数切换五种场景：正常/空/错误/边界/权限异常。

---

## 6. 测试滞后或缺失

**症状**：代码写完了但没有测试，或者测试在代码之后很久才补。

**后果**：无法验证代码是否满足验收标准；后续迭代可能破坏已有功能。

**恢复**：测试必须与代码同步产出。如果跳过了测试，回到阶段 7 补充：每个功能节点的验收标准至少映射一个测试用例，四种状态各至少一个测试。运行 `npm run test` 确认全部通过后才算完成。

---

## 7. API 层与 UI 层耦合

**症状**：组件中直接调用 `fetch()` 或 `axios`，没有通过统一的 API 层和 adapter。

**后果**：切换 mock 和真实后端需要修改多个文件；API 变更时影响面大。

**恢复**：所有数据获取走 `src/api/` 层，组件通过 hooks 消费数据。Mock ↔ 真实后端的切换只改 `src/api/client.ts` 中的 baseURL 或 interceptor。检查是否有组件直接 import `fetch` 或裸调 HTTP 库。

---

## 8. 重复造轮子

**症状**：迭代模式下新建了项目中已存在的组件、hooks、types、utils，或用到了项目已有依赖的功能但自己手写了一遍。

**后果**：代码库中出现功能重复的组件（如两个不同版本的 `UserAvatar`），类型定义不统一，维护成本翻倍。

**恢复**：
1. 在 `specs/project-context.md` 中维护一份"可复用资产清单"（组件、hooks、utils、types、依赖包）
2. 写任何新代码前，grep 搜索项目是否已有功能相近的实现
3. 功能相近就复用，功能不够就扩展（加 props/参数），不新建
4. 检查项目依赖中是否已有能完成此功能的包

---

## 9. 逆向基线扫描过重

**症状**：为既有项目生成 baseline 时，试图完整分析 50+ 页面的系统，花费大量时间在本次需求不涉及的页面上。

**后果**：时间浪费、上下文消耗巨大、用户等待时间长，且逆向出的大量节点标记为"inferred"未经确认。

**恢复**：
1. 按项目规模选择策略（小型全量 / 中型采样 / 大型局部）
2. 只详细逆向本次需求涉及的功能区域
3. 其余页面只记录路由和标题，标记为 `"source": "not-reverse-engineered"`
4. 对于接口动态生成按钮等复杂场景，标记 `"note": "需人工确认"` 后跳过
5. 向用户展示逆向摘要，让用户确认或修正，而非猜测

---

## 避免陷入陷阱的通用原则

1. **每个阶段有检查点** — 检查点不过不进入下一阶段
2. **用户确认关键产出** — features.json、change-summary.md、原型完成后必须用户确认
3. **自动检查优于人工检查** — 用脚本（如 delta 叠加）代替手工操作
4. **如果用户说"这个不对"，回到最早受影响的阶段重新推导** — 不修修补补，重新生成
